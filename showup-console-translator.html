What ShowUp Does

1. Reverse-lookup: which ShowUp moment maps to this console cue?
2. Activate that moment — apply its look, effects, energy level
3. Update Perform screen to highlight the active moment
4. If no mapping exists, log the event but take no action

This enables "console-driven show flow" — the LD runs cues; ShowUp follows automatically.

Translation Path by Console

Step-by-step process for each console family

ETC Eos (best)

  1. Query cue list structure via OSC
  2. Fire each cue, capture DMX via sACN
  3. Query /eos/out/get/params/{chan}
  4. Create Look with raw + approximated data
  5. Import palettes via /eos/out/get/cp/{n}
  6. Import groups via /eos/out/get/group/{n}
  7. Map cue list order → moment sequence

GrandMA3

  1. Export sequences as XML from MA3
  2. Parse cue names and structure
  3. Fire each cue via /gma3/cmd
  4. Capture DMX output via sACN
  5. Create ShowUp Look from captured DMX
  6. Import MVR for patch + groups
  7. Map sequence order → moments

ChamSys MQ

  1. Export palettes as CSV
  2. Parse → ShowUp palette colors + positions
  3. Fire each playback cue via OSC
  4. Capture DMX output via sACN
  5. Create ShowUp Look from captured DMX
  6. Import MVR for patch + groups
  7. Map playback order → moments

Onyx (via Telnet)

  1. Connect Telnet port 2323
  2. QLList → get cuelist names
  3. GQL {n} → fire each cuelist
  4. Capture DMX output via sACN
  5. Label looks with cuelist names
  6. Manual patch or CSV import
  7. QLActive → auto-detect changes

What Can't Be Translated

Console features with no ShowUp equivalent

Console FeatureWhy It Can't Translate
Tracking changesConsole cues store only changes from previous cue. ShowUp looks are absolute snapshots. Must fire cues in sequence to get cumulative state.
Multi-part cuesConsole cues can have parts that fire at different times. ShowUp moments are atomic.
Effects (chases)Console effects are parameter-based with step tables. ShowUp effects are procedural algorithms. Similar appearance, different data.
Fixture-level timingConsole cues can have per-fixture fade times. ShowUp applies uniform fades.
Programmer stateConsole's live adjustments (not yet recorded) have no ShowUp equivalent beyond locked channels.
Conditional cuesConsole cue lists can have follow/wait/halt triggers. ShowUp moments are operator-driven.

The Pragmatic Approach

Capture + Map instead of perfect translation:

  1. Capture the console's DMX output for each cue → raw ShowUp looks
  2. Map the cue list structure → ShowUp moment sequence
  3. Import palette data where available → ShowUp colors
  4. Leave the rest to the operator — ShowUp's reactive effects add value precisely because they're different

Universal Fallback: DMX Sniffing

Works with every console on every network

How It Works

ShowUp listens for sACN multicast (port 5568). Using the known fixture patch, it maps raw DMX values back to fixture parameters in real-time.

Fixture ParameterDMX Channel TypeInference Quality
IntensityDimmerExact
Color (RGB)Red / Green / BlueExact
Color (CMY)Cyan / Magenta / YellowExact (inverted)
Color wheelColor wheel channelExact (with wheel def)
Pan / TiltPan / Tilt channelsExact
Gobo selectionGobo wheel channelExact (with wheel def)
Zoom / Focus / IrisBeam channelsExact
Strobe rateStrobe channelApproximate
Effect speedSpeed channelApproximate

What DMX Sniffing Cannot Tell ShowUp

  • Which cue is active (just the resulting output)
  • Cue names or metadata
  • Fade times (only the current state, not the trajectory)
  • Whether output is from a cue, programmer, or effect
  • Console UI state (which fader the LD is touching)
